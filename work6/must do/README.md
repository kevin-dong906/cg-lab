## 一、实验目的

3. **数值积分方法对比**：独立实现显式欧拉、半隐式欧拉和隐式欧拉三种求解器，观察比较其稳定性差异。

## 二、实验原理

### 2.1 质点-弹簧模型

布料离散化为 `N×N` 个质点，相邻质点通过弹簧连接（基础实现仅包含结构弹簧）。每个质点具有位置 `x`、速度 `v` 和质量 `m`，受力包括：

- **重力**：`G = m * g`（竖直向下）
- **阻尼力**：`F_d = -k_d * v`，用于耗散能量，防止震荡发散
- **弹簧力**：根据胡克定律，质点 a 与 b 之间的弹力为：
```
F_spring = -k_s * (|x_a - x_b| - l) * (x_a - x_b) / |x_a - x_b|
```
其中 `k_s` 为劲度系数，`l` 为弹簧原长。该力使两质点趋向于保持原长距离。

### 2.2 数值积分方法

根据牛顿第二定律，加速度 `a = F/m`。通过离散时间步长 `dt` 更新速度和位置。本实验比较三种方法：

| 方法 | 速度更新 | 位置更新 | 稳定性 | 说明 |
| :--- | :--- | :--- | :---: | :--- |
| 显式欧拉 | `v_{t+1} = v_t + a_t * dt` | `x_{t+1} = x_t + v_t * dt` | 极易发散 | 完全使用旧状态，能量不守恒 |
| 半隐式欧拉 | `v_{t+1} = v_t + a_t * dt` | `x_{t+1} = x_t + v_{t+1} * dt` | 较稳定 | 先更新速度，再用新速度更新位置，能量略有衰减 |
| 隐式欧拉 | `v_{t+1} = v_t + a_{t+1} * dt` | `x_{t+1} = x_t + v_{t+1} * dt` | 最稳定（阻尼大） | 使用未来时刻的力，需迭代求解（本实验定点迭代3次） |

### 2.3 GPU 并行与性能优化

- **`@ti.kernel`**：用于定义并行计算函数，循环自动映射到 GPU 线程。
- **`@ti.func`**：定义内联函数，编译时展开，减少函数调用开销。
- **Kernel 拆分与合并**：为保持 GPU 状态同步，将初始化拆分为多个 kernel；为减少启动开销，将力计算与积分合并为一个 kernel（每帧仅启动一次）。

## 三、实验任务与实现

### 3.1 任务1：场景初始化（GPU 同步）

**目标**：构建 `N×N` 网格布料，初始化位置、速度、固定状态，并建立弹簧拓扑。

**关键数据结构**（Taichi Field）：

| 字段 | 类型 | 用途 |
| :--- | :--- | :--- |
| `x` | `ti.Vector.field(3, float)` | 质点位置 |
| `v` | `ti.Vector.field(3, float)` | 质点速度 |
| `f` | `ti.Vector.field(3, float)` | 当前受力（显式/半隐式） |
| `is_fixed` | `ti.field(int)` | 固定点标记（0/1） |
| `spring_pairs` | `ti.Vector.field(2, int)` | 弹簧两端索引 |
| `spring_lengths` | `ti.field(float)` | 弹簧原长 |
| `num_springs` | `ti.field(int)` | 弹簧总数 |

**初始化拆分为三个 kernel**：
- `init_positions()`：设置所有质点位置、速度、固定状态。
- `init_springs()`：遍历网格建立结构弹簧（右、下两个方向），使用 `ti.atomic_add` 安全增加弹簧计数。
- `init_spring_indices()`：生成用于 `scene.lines()` 渲染的线性索引数组。

**Python 层顺序调用**确保 GPU 同步（kernel 执行是串行提交的，但每个 kernel 内部并行）。参考代码：
```python
def init_cloth():
    num_springs[None] = 0
    init_positions()
    init_springs()
    init_spring_indices()
```

> **关键点**：多个 kernel 之间自动同步，无需额外屏障。

### 3.2 任务2：力学计算与防爆处理

**力计算函数 `compute_forces_on(pos, vel, force)`**（以 `@ti.func` 声明，内联至调用 kernel）：

1. 第一遍循环：清空 `force`，施加重力和阻尼力（每个质点独立，无竞争）。
2. 第二遍循环：遍历所有弹簧，计算弹力，使用 `ti.atomic_add` 累加到两端质点（避免多线程写冲突）。

**速度钳制 `clamp_velocity(vel, idx)`**（内联函数）：
```python
if vel_norm > max_velocity:
    vel[idx] = vel[idx] / vel_norm * max_velocity
```
防止数值爆炸导致无限大速度。

### 3.3 任务3：三种积分求解器

每个求解器均为一个 `@ti.kernel`，内部调用 `compute_forces_on` 并更新位置/速度。注意所有操作合并到一个 kernel 中，每帧仅启动一次（40 个子步，但 `step_xxx` 内包含多个 for 循环，仍为一个 kernel）。

**显式欧拉**（`step_explicit`）：
```
compute_forces_on(x, v, f)
for i: if not fixed:
    x[i] += v[i] * dt
    v[i] += (f[i]/mass) * dt
    clamp_velocity(v,i)
```
稳定性最差，易发散。

**半隐式欧拉**（`step_semi_implicit`）：
```
compute_forces_on(x, v, f)
for i: if not fixed:
    v[i] += (f[i]/mass) * dt
    clamp_velocity(v,i)
    x[i] += v[i] * dt
```
先更新速度，再用新速度更新位置，稳定性较好。

**隐式欧拉**（`step_implicit_iter`）：
使用定点迭代法近似求解未来状态。步骤：
1. 复制当前状态到 `x_next, v_next`。
2. 进行 3 次迭代（`ti.static(range(3))` 编译期展开）：
   - 计算 `f_next`（基于 `x_next, v_next`）
   - 更新 `v_next` = `v[i] + (f_next[i]/mass)*dt`
   - 更新 `x_next` = `x[i] + v_next[i]*dt`
3. 将收敛结果写回 `v, x`。

由于隐式欧拉使用未来力，更稳定，但计算量稍大（多次迭代）。

### 3.4 任务4：渲染与 GGUI 交互

使用 `ti.ui.Window` 创建窗口，`scene` 绘制粒子点和弹簧线。

**控制面板**（`window.GUI`）：
- 三个按钮切换积分方法（带 `[*]` 标记表示当前方法）。
- 暂停/继续按钮。
- 重置按钮（调用 `init_cloth()`）。

**渲染循环**：
- 每帧执行 40 个子步（总时间 40×5e-4 = 0.02s，约 50 FPS）。
- 调用 `scene.particles()` 和 `scene.lines()` 绘制布料。
- 通过 `camera.track_user_inputs()` 支持鼠标交互旋转视角。

## 四、核心代码深度分析

### 4.1 弹簧索引构建

在 `init_springs` 中，使用 `ti.atomic_add` 安全递增弹簧计数器：
```python
c = ti.atomic_add(num_springs[None], 1)
spring_pairs[c] = ti.Vector([idx, idx_right])
spring_lengths[c] = (x[idx] - x[idx_right]).norm()
```
因多线程同时写入 `num_springs`，必须原子操作避免计数丢失。

### 4.2 力计算中的原子累加

在 `compute_forces_on` 第二遍循环，使用 `ti.atomic_add` 将弹簧力累加到两端质点的 `force` 场：
```python
ti.atomic_add(force[idx_a], f_spring)
ti.atomic_add(force[idx_b], -f_spring)
```
因为多个弹簧可能同时影响同一质点，累加必须原子化，防止数据竞争。

### 4.3 Kernel 内循环顺序执行

在 `compute_forces_on` 中，两个 `for` 循环是顺序执行的（GPU 保证同一 kernel 内的不同循环按序执行），因此清空受力与累加弹簧力不会相互干扰。

### 4.4 隐式欧拉的定点迭代

`ti.static(range(3))` 在编译期展开为 3 次循环体，不引入运行时循环开销。每次迭代使用更新后的 `x_next, v_next` 重新计算力，逐步逼近隐式解。3 次迭代已能提供足够稳定性。

## 五、运行结果与对比分析

### 5.1 三种积分方法稳定性对比

| 积分方法 | 布料行为 | 稳定性 | 适用场景 |
| :--- | :--- | :---: | :--- |
| 显式欧拉 | 迅速震荡爆炸，质点飞出 | 极差 | 不推荐 |
| 半隐式欧拉 | 布料自然下垂，轻微摆动后趋于稳定 | 良好 | 实时模拟首选 |
| 隐式欧拉 | 布料下沉缓慢，几乎无振动，呈现明显阻尼 | 优秀 | 需要快速稳定的场景 |

**关键观察**：
- 显式欧拉因完全使用旧速度更新位置，系统能量不断增加，很快发散。
- 半隐式欧拉引入隐式速度更新，能量略有耗散，适合大多数布料的实时模拟。
- 隐式欧拉通过迭代求解，具有“强阻尼”特性，运动迟钝但极为稳定，适合刚性较大或需要快速收敛的情况。

### 5.2 阻尼系数 `k_d` 的影响

| 阻尼值 | 视觉效果 |
| :---: | :--- |
| 0.1 | 布料抖动剧烈，几乎无衰减 |
| 1.0 | 正常阻尼，摆动逐渐停止 |
| 5.0 | 强烈阻尼，布料几乎瞬间静止 |

### 5.3 性能表现

- 使用 GPU 并行，20×20 网格（400 质点，约 760 弹簧）每帧 40 子步，帧率稳定 >60 FPS。
- 隐式欧拉因多 3 次迭代，计算量稍大，但仍保持实时。

## 六、常见问题与解决方案

| 问题现象 | 可能原因 | 解决方法 |
| :--- | :--- | :--- |
| 布料爆炸飞走 | 时间步长过大或刚度系数过高 | 减小 `dt` 或降低 `k_s`，增加阻尼 |
| 布料的角点不固定 | `is_fixed` 判断条件错误 | 检查初始化中固定点的索引条件 |
| 弹簧绘制缺失或错位 | `spring_indices` 数组初始化未正确执行 | 确保 `init_spring_indices` 被调用 |
| 切换方法后布料撕裂 | 未重置初始状态 | 切换时调用 `init_cloth()` |
| 力计算无累加效果 | 未使用 `ti.atomic_add` | 确认累加时使用原子操作 |
| 隐式欧拉布料穿透地面 | 缺少碰撞检测 | 选做内容，可使用简单位置钳制 |

## 七、实验总结

本次实验完整实现了基于质点-弹簧模型的布料模拟系统，核心收获包括：

- **物理模拟基础**：深入理解了胡克定律、阻尼力在布模拟中的作用，以及三种数值积分方法的稳定性差异。
- **GPU 编程实践**：掌握了 Taichi 中 `ti.kernel`、`ti.func` 的使用，以及通过 kernel 拆分保证状态同步、合并 kernel 减少启动开销的性能优化技巧。
- **交互式渲染**：熟悉了 GGUI 的构建流程，实现了实时控制面板（方法切换、暂停、重置）。
- **数值稳定性认知**：通过直观对比显式/半隐式/隐式欧拉，深刻体会到半隐式欧拉在精度与稳定性之间的优秀平衡，是实时物理模拟的实用选择。

本实验为后续更复杂的物理模拟（如弹性体、流体、碰撞处理）奠定了坚实的基础。
